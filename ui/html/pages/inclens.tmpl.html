IncLens

A terminal-based C++ include inspector that reads a preprocessed .ii file and visualizes include relationships as either a navigable tree or a flamegraph.

view source

stack: Go / Bubble Tea / C++ include analysis

Overview

IncLens takes something most people never want to read directly, a preprocessed C++ source file, and turns it into an interface you can actually navigate.

It is built around a very practical workflow: generate a preprocessed translation unit with g++, feed the resulting .ii file into the tool, and then inspect the include structure from the terminal instead of scrolling through raw preprocessor output.

The value is straightforward. It helps make dependency structure visible, surfaces heavy includes earlier, and gives a faster path to compile-time investigation when the include graph starts getting harder to reason about by inspection.

The input model

The whole tool is built around the idea that the preprocessed file already contains the structure you need, but not in a form that is pleasant to inspect.

A .ii file is the fully preprocessed C++ source after #include expansion and macro processing. IncLens treats that as a data source for include analysis instead of as something meant for humans to read line by line.

That makes the tool feel focused. It is not trying to be a general-purpose IDE, compiler frontend, or indexing service. It does one thing: turn preprocessor output into something navigable and useful.

Two views of the same problem

The README sets the product up around two complementary views. That is the right design choice here, because include structure has two different questions behind it: what includes what, and where is the weight actually concentrated.

Top-down tree view

The tree view is the structural mode. This is the one you use when you want to walk the include hierarchy in a way that still feels close to the source-level mental model.

  • Navigate the include hierarchy as a tree.
  • Expand and collapse nodes with keyboard controls.
  • Search includes and jump through matches.
  • Show or hide columns and sort by bytes or lines of code.

The useful detail is that this is not just a raw tree dump. The sorting controls and file metrics make it a diagnostic view, not just a visualization.

Flamegraph view

The flamegraph mode is the proportional mode. Instead of asking for exact structure first, it asks where the code volume is concentrated and how large different parts of the include tree actually are.

  • Visualize total lines of code proportionally.
  • Move through the graph with arrow keys.
  • Zoom in with Enter and back out with Esc.
  • Inspect the current file name and line count in the info panel.

That pairing works well. The tree helps with orientation, and the flamegraph helps with weight.

Workflow

The user flow is intentionally small, which is part of the appeal. There is no large setup story and no complicated project indexing step in the README. You prepare the input once, then run the tool.

  • Generate a preprocessed .ii file with g++.
  • Run IncLens against that file.
  • Switch between the tree view and flamegraph depending on whether you want structure or proportional weight.
g++ -E -o example.ii example.cpp
./inclensapp -file example.ii

The installation story in the README is similarly light: clone the repository, run ./setup.sh, and use Go 1.20+.

Interaction details that make it better

The project gets more convincing once you look at the small controls rather than just the headline features.

  • Search and next-match navigation matter because real include graphs get wide quickly.
  • Expand-all and collapse-all help when you need to reset your view after drilling in.
  • Column toggling and sort options turn the tree into a usable investigation surface instead of a static diagram.
  • Keyboard-driven zoom in the flamegraph keeps the whole experience inside the terminal workflow.

Why it works well as a project

What I like most about it is that the scope is narrow, the workflow is clear, and the output is genuinely useful if you work in C++.

  • It takes a noisy artifact and turns it into something readable.
  • It stays in the terminal, which fits naturally into a C++ workflow.
  • It focuses on a narrow problem and gives two useful views of the same data.
  • It is a good example of a tool that feels small in interface but still solves a real engineering pain point.
