Monkey in C++

A C++ implementation of the Monkey programming language from Writing an Interpreter in Go, built as a tree-walking interpreter with a lexer, Pratt parser, explicit AST, evaluator, small REPL, and Catch2-based test suite.

Overview

This repository follows the same general path as Thorsten Ball’s Monkey interpreter, but the implementation is written in modern C++ and extends the runtime side of the project quite a bit beyond the tutorial baseline.

At a high level, it is a tree-walking interpreter with a lexer, a Pratt parser, an explicit AST, an evaluator that walks that AST, a small REPL, and a Catch2-based test suite.

The README also makes the project’s direction clear: this is still an interpreter rather than a compiler or VM, but it includes a substantial amount of work around symbol resolution, runtime values, temporary versus persistent storage, and profiling-driven performance improvements.

What it is

Monkey is a small dynamically typed language, and this implementation covers the pieces you would expect from that style of interpreter project.

  • Integers, booleans, prefix and infix expressions.
  • if / else, let, and return.
  • Functions and closures.
  • Strings, arrays, hashes, indexing, and builtins like len, first, last, rest, and push.

That language surface is broad enough that the project quickly stops being only about parsing syntax. Once closures, arrays, hashes, and recursive calls are part of the runtime, the data model and execution path start to matter a lot more.

Technical overview

The parser is a Pratt parser, which means expression parsing is driven by token precedence and small prefix and infix parse functions instead of a large grammar table.

1 + 2 * 3
add(1, 2, 3)
array[1 + 1]

The parser produces an AST, and evaluation is then done by recursively walking that tree to produce runtime values. The README breaks the full flow into four steps.

  1. The lexer turns source text into tokens.
  2. The Pratt parser turns tokens into an AST.
  3. A resolver assigns indexed symbols for globals, locals, and builtins.
  4. The evaluator walks the AST and computes results.

Very brief AST theory

The AST section in the README is useful because it explains why parsing matters beyond just turning text into nodes. The tree shape is what preserves meaning.

1 + 2 * 3
(1 + (2 * 3))

Instead of working on raw text, the evaluator gets a structure where precedence is already encoded. That is what lets evaluation operate semantically rather than textually.

Example program

The README uses recursive Fibonacci as the main benchmark example, but it is also a useful demonstration that the language semantics work end to end.

let fibonacci = fn(x) {
  if (x == 0) {
    0
  } else {
    if (x == 1) {
      1
    } else {
      fibonacci(x - 1) + fibonacci(x - 2);
    }
  }
};

fibonacci(10);

Performance notes

This is still an AST interpreter, not a compiler or VM. Even so, the README shows a clear line of performance work and local measurements that improved over time.

The current optimized fibonacci(33) benchmark is described in the README as roughly ~1.3s - 1.6s wall-clock with peak memory around ~1.6MB - 1.8MB on the local machine.

  • Local fibonacci(33) measurements are around ~1.3s - 1.6s with the optimized preset.
  • Peak memory is reported around ~1.6MB - 1.8MB.
  • The main remaining bottleneck is still the nature of recursive AST evaluation, not a single obvious memory-management issue.

Those figures are described as local measurements rather than a formal benchmark suite, which is a sensible distinction to keep.

Profiling observations

The optimization work is organized around repeated profiling of the fibonacci(33) benchmark. The sequence in the README is one of the stronger parts of the project because it shows how the bottleneck moved as each layer improved.

  • Early versions spent a lot of time in AST dispatch and RTTI-based type checks.
  • After that improved, runtime name lookup became a visible cost.
  • After lookup improved, object allocation and ownership churn became the dominant issue.
  • After the Value and arena refactor, over-persisting recursive call frames became the main memory issue.
  • After temporary call-frame separation, the remaining cost was mostly recursive AST evaluation with smaller overhead from arguments and general interpreter machinery.

Refactorings and their effect

1. Removing expensive AST dispatch

The first optimization step was moving away from heavier runtime dispatch and RTTI-style checks in the evaluator.

  • AST nodes carry an explicit NodeType.
  • Evaluator logic dispatches by node kind.
  • Cloning and evaluation avoid more expensive runtime type paths.

2. Indexed symbol resolution

Instead of resolving names entirely through string-based environments at runtime, the interpreter assigns indexed references for globals, locals, and builtins ahead of time.

  • Fewer string operations in the hot path.
  • Cheaper local and global access.
  • Better scaling for deeply recursive programs.

3. Value as a tagged runtime handle

The runtime model uses immediate representations for simple values and heap-backed objects only where that is actually needed.

  • int is immediate.
  • bool is immediate.
  • null is immediate.
  • Composite values remain heap-backed.

4. Scratch arena and persistent store

Composite values do not all share the same lifetime, so the runtime separates temporary work from values that must escape.

  • ScratchArena handles temporary composites created during evaluation.
  • PersistentStore keeps values that must outlive the current scratch lifetime.

5. Explicit promotion and detachment

The runtime makes lifetime transitions explicit through promote(Value, PersistentStore&) and detach(Value).

  • Temporary values stay temporary by default.
  • Escaping values are promoted deliberately.
  • Final top-level results can be detached safely.

6. Temporary call frames instead of persistent recursive frames

The README highlights this as one of the most important fixes. Ordinary recursive calls should not hold onto persistent lifetime unless something really escapes.

  • Normal calls use temporary stack-like Environment frames.
  • Persistent storage is reserved for closures or returned values that need to survive.

7. Small-arity argument fast path

After the larger lifetime issues were addressed, argument handling still showed up in profiles.

  • Common cases like 0, 1, and 2 arguments avoid allocating a std::vector<Value>.
  • This is especially helpful for recursive benchmark shapes like Fibonacci.

8. Build-level optimizations

The repository also supports stronger optimized builds through CMake presets, including release, release-lto, and release-native-lto.

  • The fastest preset combines release optimization, link-time optimization, and native CPU tuning.
  • The README explicitly notes that benchmark numbers should be read together with the chosen build preset.

Current remaining bottlenecks

The README is clear that the main remaining cost is no longer memory management itself but the nature of the interpreter.

  • Recursive AST evaluation.
  • Repeated tree-walking through eval_node.
  • Ordinary interpreter overhead around calls, branching, and expression evaluation.

The suggested next major step in the README is some form of compilation or VM rather than another large AST-evaluator refactor.

Inspiration

The project follows the spirit and structure of Writing an Interpreter in Go, so if you know the book the overall shape is familiar: tokens, lexer, Pratt parser, AST, evaluator, environments, and closures.

The main difference is the implementation language and the amount of runtime optimization work layered on top of the basic interpreter design.
